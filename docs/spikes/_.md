# Plan: WebSocket Platform-as-a-Service (PaaS) Transformation

## Overview

Transform the existing high-performance WebSocket server into a multi-tenant Platform-as-a-Service offering where users can spawn dedicated instances and configure them via a Web UI.

**Architecture Decisions:**
- Dedicated Kubernetes instances per tenant (not shared infrastructure)
- Kafka-only message source
- Initial use cases: chat apps, real-time UI updates, trading platforms

## 3. Changes Required to WebSocket Server

### 3.1 Dynamic Topic Configuration (Currently Hardcoded)

**Current:** `kafka/config.go` has 8 hardcoded topics
**Required:** Load topics from config file or environment variable

```yaml
# /config/tenant.yaml
kafka:
  topics:
    - name: "acme.trades"
      event_type: "trade"
      enabled: true
    - name: "acme.orders"
      event_type: "order"
      enabled: true
```

**Files to modify:**
- `internal/shared/kafka/config.go` - Create TopicRegistry
- `internal/shared/kafka/consumer.go` - Use dynamic topics
- `internal/shared/platform/config.go` - Add topics config loading

### 3.2 Configurable Channel Patterns

**Current:** `broadcast.go` expects `sukko.token.{SYMBOL}.{EVENT}`
**Required:** Configurable pattern per tenant

```go
type ChannelExtractor interface {
    Extract(subject string) string
}
```

**Files to modify:**
- `internal/shared/broadcast.go` - Use ChannelExtractor interface

### 3.3 API Key Authentication

**Current:** No authentication on WebSocket endpoint
**Required:** API key validation before WebSocket upgrade

```go
// New: internal/shared/auth/authenticator.go
type Authenticator interface {
    Authenticate(r *http.Request) (TenantContext, error)
}
```

**Files to modify:**
- `internal/shared/handlers_ws.go` - Add authentication check

### 3.4 Tenant Config File Schema

```yaml
# /config/tenant.yaml - Complete tenant configuration

# Kafka topic configuration
kafka:
  topics:
    - name: "myapp.orders"          # Kafka topic name
      event_type: "order"           # Maps to channel suffix
      enabled: true
    - name: "myapp.notifications"
      event_type: "notify"
      enabled: true

# Generic channel pattern configuration
channel:
  # Pattern with placeholders extracted from Kafka message
  # Available placeholders: {key}, {topic}, {event_type}, {partition}
  # The {key} is the Kafka message key (e.g., user ID, room ID, device ID)
  pattern: "{key}.{event_type}"     # e.g., "user123.order", "room456.notify"
  separator: "."

  # Alternative patterns for different use cases:
  # Chat:     "{key}.{event_type}"  → room123.message, room123.typing
  # Trading:  "{key}.{event_type}"  → BTC.trade, ETH.order
  # IoT:      "{key}.{event_type}"  → sensor001.temperature
  # Custom:   "{topic}.{key}"       → myapp.orders.user123

# Resource limits
limits:
  max_connections: 5000
  max_kafka_rate: 1000          # msg/sec from Kafka
  max_broadcast_rate: 50        # broadcasts/sec
  client_buffer_size: 512       # slots per client

# Authentication
auth:
  enabled: true
  mode: "api_key"               # api_key | jwt | custom_header
  header_name: "X-API-Key"
  query_param: "api_key"        # Also check query string
  keys:                         # For Phase 1 (local validation)
    - "sk_live_abc123..."
    - "sk_test_xyz789..."

# Logging
logging:
  level: "info"                 # debug | info | warn | error
  format: "json"                # json | text
```

## 4. Control Plane Components

### 4.1 API Server Endpoints

```
/api/v1/tenants
  POST   /                  - Create tenant
  GET    /:id               - Get tenant
  PUT    /:id               - Update tenant
  DELETE /:id               - Deprovision

/api/v1/tenants/:id/config
  GET    /                  - Get config
  PUT    /                  - Update config

/api/v1/tenants/:id/topics
  GET    /                  - List topics
  POST   /                  - Add topic
  DELETE /:topic            - Remove topic

/api/v1/tenants/:id/api-keys
  POST   /                  - Generate key
  DELETE /:keyId            - Revoke key

/api/v1/tenants/:id/metrics
  GET    /connections       - Connection stats
  GET    /messages          - Throughput
```

### 4.2 PostgreSQL Schema

```sql
CREATE TABLE tenants (
    id UUID PRIMARY KEY,
    name VARCHAR(100) UNIQUE,
    slug VARCHAR(50) UNIQUE,
    plan_id UUID REFERENCES plans(id),
    status VARCHAR(20) DEFAULT 'pending'
);

CREATE TABLE tenant_topics (
    id UUID PRIMARY KEY,
    tenant_id UUID REFERENCES tenants(id),
    kafka_topic VARCHAR(200),
    event_type VARCHAR(50),
    enabled BOOLEAN DEFAULT true
);

CREATE TABLE api_keys (
    id UUID PRIMARY KEY,
    tenant_id UUID REFERENCES tenants(id),
    key_hash VARCHAR(64),
    key_prefix VARCHAR(8),
    expires_at TIMESTAMP
);
```

### 4.3 Kubernetes Operator

- Watch Tenant CRDs
- Create per-tenant: Namespace, Deployment, ConfigMap, Service, Ingress, HPA, NetworkPolicy
- Handle config updates (rolling restart)

## 6. Implementation Phases

### Phase 1: Core Flexibility (CURRENT FOCUS)
Make WebSocket server fully configurable via files/env vars

#### 1.1 Dynamic Topic Configuration
- [ ] Create `TopicRegistry` struct in `kafka/config.go`
- [ ] Support loading topics from YAML config file
- [ ] Support `KAFKA_TOPICS` env var (JSON format)
- [ ] Remove hardcoded topic constants
- [ ] Update consumer to use dynamic topics

#### 1.2 Generic Channel Pattern System
- [ ] Create `ChannelExtractor` interface
- [ ] Implement configurable pattern matching with placeholders
- [ ] Support patterns like: `{entity}.{event}`, `{prefix}.{id}.{type}`, etc.
- [ ] Add pattern validation at startup
- [ ] Update `broadcast.go` to use configurable extractor

#### 1.3 API Key Authentication
- [ ] Create `internal/shared/auth/` package
- [ ] Implement `APIKeyAuthenticator` with header/query support

- [ ] Create YAML schema for tenant configuration
- [ ] Add config file loader with validation
- [ ] Support hot-reload via fsnotify (optional)
- [ ] Create example config file

## 10. Decisions Made

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Target use cases | Generic (all patterns) | Maximum flexibility for unknown use cases |
| Initial scope | Phase 1 only | Validate core flexibility before building control plane |
| Control plane framework | Go + Fiber | User preference; Express-like API, fast |
| Tenant isolation | Dedicated K8s instances | Simpler security, easier debugging |
| Message source | Kafka only | Current architecture; no need for complexity |
