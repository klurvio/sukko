## 1. Prerequisites

### 1.1 Setup sequence (ORDER MATTERS)

```
sukko auth keygen        # MUST run before sukko up
sukko up                 # starts the stack
sukko health             # wait for green
sukko status             # confirm all services running
sukko edition            # capture edition + license key path
sukko test smoke         # final gate before running suite battery
```

> **Ordering constraint**: If you run `sukko up` before `sukko auth keygen`,
> provisioning starts without `ADMIN_BOOTSTRAP_KEY` and all admin-keypair suites
> (`provisioning`, `tenant-isolation`, `token-revocation`) fail at setup with no
> obvious error.

### 1.2 TESTER_ADMIN_KEY_FILE format note

The `TESTER_ADMIN_KEY_FILE` env var expects a **raw 64-byte Ed25519 binary private
key** — NOT a PEM-encoded PKCS#8 key. The common failure mode is exporting a key
from OpenSSL as PKCS#8 PEM and supplying that file, which causes a parse error with
no helpful message at startup.

Correct format (generated by `sukko auth keygen`):
```
64 raw bytes: 32-byte seed || 32-byte public key
```

To verify your key file is the right format:
```bash
wc -c ~/.sukko/admin.key   # should print exactly 64
```

### 1.3 Required environment variables

| Variable | Default | Required | Description |
|---|---|---|---|
| `GATEWAY_URL` | `ws://localhost:3000` | always | WebSocket gateway address |
| `PROVISIONING_URL` | `http://localhost:8080` | always | Provisioning API address |
| `TESTER_AUTH_TOKEN` | — | production | Token for tester HTTP API auth |
| `TESTER_ADMIN_KEY_FILE` | — | provisioning/tenant-isolation/token-revocation suites | Raw 64-byte Ed25519 admin private key |
| `TESTER_ADMIN_KEY_ID` | `bootstrap-0` | when admin key file is set | Key ID registered in provisioning |
| `TESTER_JWT_LIFETIME` | `15m` | no | JWT lifetime issued by tester |
| `TESTER_JWT_REFRESH_BEFORE` | `2m` | no | Auth refresh trigger threshold |
| `TESTER_KEY_EXPIRY` | `24h` | no | Signing key expiry issued by tester |
| `TESTER_PORT` | `8090` | no | Tester HTTP server port |
| `PROVISIONING_TOKEN_REVOCATION_MAX_LIFETIME` | `1h` | no | Max TTL for revoked token entries (provisioning service) |

## 2. Validate Suite Battery

Run all 14 suites in the order below. Each suite is independent and can be run
individually with `sukko test validate --suite <name>`.

Signal vocabulary:
- `Report.Status`: `pass` / `fail` / `error`
- `CheckResult.Status`: `pass` / `fail` / `skip`

### 2.1 Suite run order

| # | Suite | Min Edition | Admin Key? | Description |
|---|---|---|---|---|
| 1 | `smoke` | Community | no | Basic connectivity gate |
| 2 | `auth` | Community | no | JWT auth validation |
| 3 | `channels` | Community | no | Channel subscribe/unsubscribe |
| 4 | `pubsub` | Community | no | Pub-sub delivery with scoping |
| 5 | `ordering` | Community | no | Message FIFO ordering |
| 6 | `reconnect` | Community | no | Disconnect/reconnect recovery |
| 7 | `ratelimit` | Community | no | Rate limit enforcement |
| 8 | `sse` | Community | no | SSE transport validation |
| 9 | `rest-publish` | Community | no | REST publish delivery |
| 10 | `provisioning` | Community | **yes** | Provisioning API CRUD |
| 11 | `tenant-isolation` | Community | **yes** | Cross-tenant isolation |
| 12 | `token-revocation` | **Pro** | **yes** | Token revocation force-disconnect |
| 13 | `edition-limits` | Community | no | Edition boundary limits |
| 14 | `push` | Enterprise | no | Push notification pipeline |
| 15 | `license-reload` | Community | no | License hot-reload propagation |

> Note: `edition-limits` does NOT test `MaxTopicsPerTenant` — it is excluded from
> the normative test matrix. Do not add it as a tested dimension.

### 2.2 smoke

```bash
sukko test validate --suite smoke
```

Validates basic connectivity: HTTP health check, WebSocket upgrade handshake, and
tester API reachability. This is the mandatory gate before running any other suite.

**Minimum edition**: Community  
**Admin key**: not required

**Expected passing output**:
```
Suite: smoke  Status: pass
  [pass] health_check
  [pass] ws_connect
```

### 2.3 auth

```bash
sukko test validate --suite auth
```

Validates JWT authentication: valid token accepted, expired token rejected, wrong
`kid` rejected, wrong tenant rejected, revoked key rejected, missing token rejected.

**Minimum edition**: Community  
**Admin key**: not required

**Expected passing output**:
```
Suite: auth  Status: pass
  [pass] valid_token_accepted
  [pass] expired_token_rejected
  [pass] wrong_kid_rejected
  [pass] wrong_tenant_rejected
  [pass] revoked_key_rejected
  [pass] missing_token_rejected
```

### 2.4 channels

```bash
sukko test validate --suite channels
```

Validates channel subscribe/unsubscribe lifecycle: acknowledgement messages, forced
unsubscription (gateway-initiated disconnect with `UnsubscriptionAck`), and
subscription state tracking.

**Minimum edition**: Community  
**Admin key**: not required

### 2.5 pubsub

```bash
sukko test validate --suite pubsub
```

Validates publish-subscribe delivery with channel scoping: public channels, user
channels (`user.<sub>`), and group channels. Confirms cross-tenant isolation at the
delivery layer.

**Minimum edition**: Community  
**Admin key**: not required

### 2.6 ordering

```bash
sukko test validate --suite ordering
```

Validates FIFO message ordering across a single partition. Publishes a sequence of
numbered messages and asserts they arrive in order.

**Minimum edition**: Community  
**Admin key**: not required

### 2.7 reconnect

```bash

```

Validates session recovery after disconnect: reconnect with a new JWT, subscription

**Minimum edition**: Community  
**Admin key**: not required

**Minimum edition**: Community  
**Admin key**: not required

### 2.10 rest-publish

```bash
sukko test validate --suite rest-publish
```

Validates the REST publish endpoint (`POST /api/v1/tenants/{slug}/publish`): message

**Minimum edition**: Community  
**Admin key**: not required

### 2.11 provisioning

```bash
sukko test validate --suite provisioning
```

Validates the provisioning API CRUD lifecycle: tenant create/get/update/list, key
create/revoke, API key create/revoke, routing rules create/list/delete. Requires an
admin keypair — see §1.2.

**Minimum edition**: Community  
**Admin key**: required (`TESTER_ADMIN_KEY_FILE` + `TESTER_ADMIN_KEY_ID`)

**Without admin key**:
```
Suite: provisioning  Status: error
  [fail] setup: RequireAdminProvider: admin key file not configured
```

### 2.12 tenant-isolation

```bash
sukko test validate --suite tenant-isolation
```

Validates cross-tenant message isolation: creates two tenants, publishes to one,
asserts the other receives nothing. Requires admin keypair.

**Minimum edition**: Community  
**Admin key**: required

### 2.13 token-revocation

```bash
sukko test validate --suite token-revocation
```

Validates token revocation behavior end-to-end. Requires Pro edition and admin
keypair.

**Scenarios covered**:
1. **Sub revocation**: revoke all tokens for a user by `sub` — connected client is
   force-disconnected with close code 4001
2. **JTI revocation**: revoke a specific token by `jti` — same close code
3. **Invalid revocations**: Community edition gate — POST to `/tokens/revoke` returns
   HTTP 403 `{"code":"EDITION_LIMIT",...}`
4. **Push cleanup** (Enterprise only): push subscription deleted on revocation

**Minimum edition**: Pro (for full pass; Community returns `pass` with one `[skip]`
on the edition-gate check — `Report.Status` is still `pass`)

**Admin key**: required

**Expected passing output (Pro)**:
```
Suite: token-revocation  Status: pass
  [pass] sub_revocation_force_disconnect
  [pass] jti_revocation_force_disconnect
  [skip] push_cleanup  (requires Enterprise)
```

**Expected output (Community — edition gate)**:
```
Suite: token-revocation  Status: pass
  [skip] sub_revocation_force_disconnect  (requires Pro)
  [skip] jti_revocation_force_disconnect  (requires Pro)
  [skip] push_cleanup  (requires Enterprise)
```

**Without admin key**:
```
Suite: token-revocation  Status: error
  [fail] setup: RequireAdminProvider: admin key file not configured
```

## 3. Load / Stress / Soak Testing

```bash
sukko test load    [--follow] [--duration=5m] [--connections=100]
sukko test stress  [--follow] [--duration=1m] [--connections=500]
sukko test soak    [--follow] [--duration=30m] [--connections=50]
```

Use `--follow` / `-f` to stream real-time SSE metrics while the test runs.

> Note: `sukko test status --id` is advertised in hint text but NOT implemented.
> Use `--follow` for live output instead.

**Key metrics to monitor during load tests**:
- `ws_server_connections_total` — connection count
- `ws_server_messages_received_total` — ingestion rate
- `ws_server_messages_sent_total` — delivery rate
- `gateway_websocket_connections_active` — gateway-side active count
- Memory growth over time (soak tests)

## 4. Manual Provisioning API Verification

All examples use `$TOKEN` (admin JWT) and `$BASE` (`http://localhost:8080`).

### 4.1 Tenant lifecycle

```bash
# Create
curl -sX POST $BASE/api/v1/tenants \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"slug":"acme","name":"Acme Corp"}' | jq

# Get
curl -s $BASE/api/v1/tenants/acme \
  -H "Authorization: Bearer $TOKEN" | jq

# List
curl -s "$BASE/api/v1/tenants?limit=20&offset=0" \
  -H "Authorization: Bearer $TOKEN" | jq

# Update
curl -sX PATCH $BASE/api/v1/tenants/acme \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"name":"Acme Corporation"}' | jq

# Suspend (409 if already suspended)
curl -sX POST $BASE/api/v1/tenants/acme/suspend \
  -H "Authorization: Bearer $TOKEN" | jq

# Reactivate
curl -sX POST $BASE/api/v1/tenants/acme/reactivate \
  -H "Authorization: Bearer $TOKEN" | jq

# Deprovision (grace period — soft delete)
curl -sX POST $BASE/api/v1/tenants/acme/deprovision \
  -H "Authorization: Bearer $TOKEN" | jq

# Force deprovision (immediate)
curl -sX POST "$BASE/api/v1/tenants/acme/deprovision?force=true" \
  -H "Authorization: Bearer $TOKEN" | jq
```

> **CLI gap**: The CLI does not yet support `sukko tenant rename` — use curl:
> ```bash
> curl -sX POST $BASE/api/v1/tenants/acme/rename \
>   -H "Authorization: Bearer $TOKEN" \
>   -H "Content-Type: application/json" \
>   -d '{"new_slug":"acme-corp"}' | jq
> ```
> (Known gap tracked for a future CLI release.)

### 4.4 Routing rules

Phase 1 constraint: only `default` and `dead-letter` system topics are valid targets
until the full channel-routing feature ships.

```bash
# Create routing rule
curl -sX POST $BASE/api/v1/tenants/acme/routing-rules \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"pattern":"trades.*","topics":["default"],"priority":10}' | jq
# Expected error if pattern not allowed: HTTP 400 ROUTING_RULE_VALIDATION_ERROR

# List
curl -s $BASE/api/v1/tenants/acme/routing-rules \
  -H "Authorization: Bearer $TOKEN" | jq

# Replace all rules
curl -sX PUT $BASE/api/v1/tenants/acme/routing-rules \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"rules":[{"pattern":"*","topics":["default"],"priority":1}]}' | jq

# Delete all rules
curl -sX DELETE $BASE/api/v1/tenants/acme/routing-rules \
  -H "Authorization: Bearer $TOKEN" | jq
```

**Error codes for routing rules**:
| Code | HTTP | Meaning |
|---|---|---|
| `TOO_MANY_ROUTING_RULES` | 400 | Edition limit exceeded |
| `ROUTING_RULE_VALIDATION_ERROR` | 400 | Pattern/topic validation failed |
| `TOPIC_NOT_PROVISIONED` | 400 | Target topic not provisioned |
| `ROUTING_RULE_DUPLICATE_PRIORITY` | 409 | Priority collision |
| `ROUTING_RULE_DUPLICATE_PATTERN` | 409 | Pattern collision |

### 4.5 Token revocation (Pro+)

```bash
# Revoke all tokens for a user by subject
curl -sX POST $BASE/api/v1/tenants/acme/tokens/revoke \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"sub":"user-123"}' | jq
# Expected: 200 {"status":"revoked","type":"user","tenant_id":"acme","expires_at":"..."}

# Revoke a specific token by JTI
curl -sX POST $BASE/api/v1/tenants/acme/tokens/revoke \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"jti":"a1b2c3d4-..."}' | jq
# Expected: 200 {"status":"revoked","type":"token","tenant_id":"acme","expires_at":"..."}

# Revoke with explicit expiry (must be in future and < PROVISIONING_TOKEN_REVOCATION_MAX_LIFETIME)
curl -sX POST $BASE/api/v1/tenants/acme/tokens/revoke \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"sub":"user-123","exp":1798761600}' | jq

# Validation error — both sub and jti
curl -sX POST $BASE/api/v1/tenants/acme/tokens/revoke \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"sub":"user-123","jti":"abc"}' | jq
# Expected: 400 {"code":"INVALID_REQUEST","message":"provide either 'sub' or 'jti', not both"}

# Edition gate (Community)
curl -sX POST $BASE/api/v1/tenants/acme/tokens/revoke \
  -H "Authorization: Bearer $COMMUNITY_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"sub":"user-123"}' | jq
# Expected: 403 {"code":"EDITION_LIMIT","message":"This feature requires pro edition or higher"}

# Rate limit (>5 requests/2s from same IP)
# Expected: 429 with Retry-After: 2 header (seconds until retry window resets)
#           {"code":"RATE_LIMITED","message":"too many revocation requests"}
```

**Error codes for token revocation**:
| Code | HTTP | Meaning |
|---|---|---|
| `INVALID_REQUEST` | 400 | Missing or conflicting fields |
| `EDITION_LIMIT` | 403 | Requires Pro edition |
| `RATE_LIMITED` | 429 | Per-IP rate limit exceeded (30 req/min); response includes `Retry-After: 2` header |
| `INTERNAL_ERROR` | 500 | Store write failure |

**Configuration**:
| Env var | Default | Description |
|---|---|---|
| `PROVISIONING_TOKEN_REVOCATION_MAX_LIFETIME` | `1h` | Max TTL for revocation entries. Zero or negative rejected at startup. Client-supplied `exp` is capped at this value. |

## 6. WebSocket Protocol Verification

### 6.1 Connect with JWT

```bash
wscat -c "ws://localhost:3000/ws?token=$JWT"
```

### 6.2 Connect with API key

```bash
wscat -c "ws://localhost:3000/ws" \
  -H "Authorization: ApiKey $API_KEY"
```

### 6.3 Subscribe and receive

After connecting, send:
```json
{"type":"subscribe","channel":"acme.trades"}
```

Expected ack:
```json
{"type":"subscription_ack","channel":"acme.trades","status":"subscribed"}
```

### 6.4 Forced unsubscription

When a tenant is suspended or a routing rule removes a channel, the gateway sends:
```json
{"type":"unsubscription_ack","channel":"acme.trades","unsubscribed":true,"forced":true}
```

### 6.5 Connection limit

Once the per-tenant connection limit is reached:
```
HTTP/1.1 429 Too Many Requests
```

### 6.6 Force-disconnect on token revocation

After calling `POST /tokens/revoke`, the connected client receives WebSocket close
code `4001` within `revocationTimeout` (default: 5 seconds).

### 6.7 Message history (Pro+)

Message history is available on Pro or higher editions with `WS_HISTORY_ENABLED=true` and a Valkey instance configured via `VALKEY_ADDRS`.

**Subscribe-with-history** — single combined request:

```json
{"type": "subscribe", "data": {"channel": "acme.BTC.trade", "history": {"limit": 50}}}
```

The server first confirms the subscription, then delivers up to 50 historical messages each tagged `"history": true`, followed by a `history_complete` signal:

```json
{"type": "broadcast", "seq": 1, "ts": 1706000000000, "channel": "acme.BTC.trade", "data": {...}, "history": true, "stream_id": "1706000000000-0"}
{"type": "history_complete", "channel": "acme.BTC.trade", "count": 50, "source": "cache", "attach_id": "1706000000000-0"}
```

**Standalone history request** — after subscribing separately:

```json
{"type": "history", "data": {"channel": "acme.BTC.trade", "limit": 50}}
```

**`attach_id` protocol** — the `attach_id` in `history_complete` marks the history/live boundary. Messages delivered after `history_complete` with no `history: true` field are live. An empty `attach_id` (`""`) means the channel has no stored history (new channel or TTL expired) — all messages are live.

**Kafka fallback** — if the Valkey ring buffer has fewer messages than `limit` (partial history), the server automatically falls back to Kafka replay to fill the gap. The `source` field in `history_complete` reflects what was actually delivered:
- `"cache"` — all messages from Valkey ring buffer
- `"mixed"` — ring buffer + Kafka fallback combined

Kafka fallback is unavailable when `BROADCAST_TYPE=valkey` (see migration notes).

**History error codes:**

| Code | Meaning | Retriable |
|------|---------|-----------|
| `history_disabled` | Feature not enabled (`WS_HISTORY_ENABLED=false`) | No |
| `not_available` | Edition gate (Pro required) or JWT expired | No |
| `history_not_subscribed` | Must subscribe before requesting history | No |
| `history_time_range_not_supported` | `from_ts`/`to_ts` not supported in v1 | No |
| `history_invalid_limit` | Limit out of range `[1, WS_HISTORY_MAX_LIMIT]` | No |
| `history_invalid_channel` | Channel not in `{tenant}.{suffix}` format | No |
| `history_invalid_request` | Malformed JSON in history request payload | No |
| `history_in_progress` | Delivery already running for this channel | Retry after `history_complete` |
| `history_storage_unavailable` | Valkey temporarily unreachable | Yes, with backoff |

## 8. Developer Accuracy Audit Checklist

Run before any release to ensure CLI output matches the code.

```bash
# 1. Enumerate all CLI commands recursively
sukko --help
sukko test --help
sukko test validate --help
sukko test load --help
sukko auth --help
sukko tenant --help
# ... repeat for all subcommands

# 2. For each command, verify:
#    a. Flags documented in --help match the code (check commands/*.go)
```
